- [ ] **Fix-artifact ↔ Verifier execution-format mismatch (surfaced post-Sprint-5 on a real `/api/v1/alerts` run)**: `CodeFixerAgent` (and `InfraSREAgent`) emit the fix as a *code fragment / function body* — often containing a bare top-level `return` — but `VerifierAgent` runs it via `client.containers.run(..., ["python", "-c", fix_artifact])`, i.e. as a **standalone script**. A bare `return` at module level is `SyntaxError: 'return' outside function`, so most code-class fixes **fail sandbox verification regardless of input quality** (reproduced on two real KeyError runs — a stub payload and the detailed `008_code_key_error_dict` fixture — both `Sandbox Execution: failure / SyntaxError`). The pipeline still completes and the report is honest ("failure"), so this is a fix-*quality*/contract gap, not a crash. **Two coupled fixes**: (1) align the contract — either CodeFixer/InfraSRE must emit a complete runnable script (no fragment / bare `return`), or `VerifierAgent` must normalize the artifact (wrap in a function, or write-to-file-and-run, or detect a fragment and adapt); (2) **tighten the benchmark `resolved` definition** in `scripts/run_benchmark.py` — it currently is `report_text is not None and execution_error is None`, where `execution_error` is set only on *docker-level* exceptions, NOT on a non-zero fix exit. So SyntaxError-failing fixes still count as "resolved", which means the reported **`resolution_rate` (e.g. 0.98 in T066) overstates real fix success** — it measures *pipeline completion*, not *fix verification*. Require `execution_result.status == "success"` (return_code 0) for a scenario to count as resolved, then re-baseline.

- [ ] **M4 mcp-enabler: `fix.code_diff` is always `""` + no real severity grading (raised on `feat/m4-mcp-enabler`)**: the `output/{stem}-result.json` sidecar (specs/m4-mcp-enabler FR-001) promises `code_diff: <unified diff>` in the MCP-facing contract, but the pipeline only ever produces an *executable fix script* (`fix_artifact`/`fix_script`), never a diff — so the sidecar honestly emits `""` and surfaces the script as `fix_plan`. Similarly, `diagnosis.severity` is a **documented deterministic fallback** (`medium`, upgraded to `high` on `security_verdict == HIGH_RISK` / `approval_required`) because no agent grades incident severity. **Resolution**: when the fix-artifact ↔ Verifier contract is reworked (entry above), have the specialist emit a structured artifact (plan + optional unified diff) and add severity to DiagnosisAgent's JSON output; then tighten the sidecar mapping and drop both fallbacks.
