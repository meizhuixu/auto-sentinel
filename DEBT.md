- [ ] **PR-5 wrap-up or Sprint 6**: `factory._load_routing_config()` reads `Path("config/model_routing.yaml")` — a path relative to the process CWD. The full test suite happens to pass because the module-level singleton cache is populated when CWD = project root; running a single test file (e.g. `pytest tests/unit/test_run_pipeline.py`) from elsewhere triggers `ConfigurationError: model routing config not found`. **Fix**: use `Path(__file__).resolve().parent.parent.parent / "config/model_routing.yaml"`.

- [ ] **T066 real-run note (raised during T056)**: the HIGH_RISK ground truth for SECURITY scenarios `030`–`035` (command injection / path traversal / hardcoded credentials / stored XSS / insecure deserialization / weak crypto) depends entirely on the **GLM SecurityReviewer's semantic judgment** at real-run time. The hard-coded `_HIGH_RISK_KEYWORDS` deny-list in `autosentinel/agents/security_reviewer.py` (DROP TABLE / rm -rf / chmod 777 / mkfs / dd if=) does **not** cover these 6 vulnerability classes, and it scans the `fix_artifact`, not the incident log — so it provides **no deterministic backstop** for them. This is intentional (Constitution V v2.2.0: LLM semantic review is the primary mechanism, deny-list is defense-in-depth only). Consequence: **SC-013 (SECURITY-subset false-negative count = 0) is a genuine test of GLM's security judgment, not a keyword guarantee.** If a T066 full run surfaces a false negative on `030`–`035`, it is a GLM-judgment problem (prompt/model tuning), NOT a benchmark-labelling problem — the labels are correct. `036` is the SAFE control (suspicious-login alert, no dangerous remediation) included to catch the opposite failure: a SecurityReviewer that blindly stamps every SECURITY-category incident HIGH_RISK.
